Phase Overview

Phase 1 establishes the monorepo foundation — a single root directory containing both the backend (Node.js + Express) and frontend (React + Vite) as separate workspaces. No application logic is written in this phase; the goal is purely to get the correct folder structure, dependencies, and tooling in place so every subsequent phase has a stable base to build on.

🗂️
Monorepo Pattern
Both backend and frontend live in one Git repository, making it easy to share code, run both servers, and deploy from a single source of truth.
⚡
Vite over CRA
Vite is chosen for the frontend because it uses native ES modules for near-instant HMR (Hot Module Replacement) during development — much faster than Create React App.
📦
ES Modules Backend
"type": "module" in package.json makes the backend use import/export syntax (ESM) instead of require/module.exports (CJS), keeping syntax consistent across the whole project.
🔍
No Nodemon
Node.js 18+ has a built-in --watch flag. We use --watch-path=./ with --env-file=.env to auto-restart and load environment variables without any extra dependencies.

Backend — package.json

This is the most important configuration file for the backend. Every field is deliberately chosen. Here is the complete annotated file:

  backend/package.json JSON 
{
  // "name" — identifies this npm package; used in logs and npm registry
  "name": "server",

  // "version" — semantic versioning; starts at 1.0.0 for a new project
  "version": "1.0.0",

  // "type": "module" — CRITICAL! Tells Node.js to use ES Module syntax.
  // This means every .js file in this project must use:
  //   import X from 'y'       ← instead of: const X = require('y')
  //   export default X         ← instead of: module.exports = X
  // Without this, Node defaults to CommonJS (require) syntax.
  "type": "module",

  // "main" — the entry point file Node.js runs when you start the server
  "main": "server.js",

  "scripts": {
    // "test" — placeholder; no tests configured yet in Phase 1
    "test": "echo \"Error: no test specified\" && exit 1",

    // "dev" — runs the server in DEVELOPMENT mode with:
    //   --env-file=.env      reads .env file and injects variables into process.env
    //                        (replaces the 'dotenv' npm package — built into Node 20.6+)
    //   --watch-path=./      watches ALL files in the backend folder for changes
    //                        and automatically restarts the server (replaces 'nodemon')
    //   server.js            the file to execute
    "dev": "node --env-file=.env --watch-path=./ server.js",

    // "start" — runs the server in PRODUCTION mode:
    //   No --watch-path because production servers should NOT auto-restart on file change
    //   --env-file=.env still loads environment variables from the .env file
    "start": "node --env-file=.env server.js"
  },

  "author": "Abhi",   // project owner name
  "license": "MIT",   // open source license type

  "dependencies": {
    // bcryptjs — hashes passwords before storing in database.
    //   Never store plain-text passwords. bcrypt adds a "salt" (random data)
    //   so even identical passwords produce different hashes.
    "bcryptjs": "^3.0.3",

    // cors — Cross-Origin Resource Sharing middleware.
    //   Browsers block requests from a different domain (e.g., localhost:5173 → localhost:5000).
    //   cors() middleware adds HTTP headers that tell browsers: "this is allowed."
    "cors": "^2.8.6",

    // express — the web framework. Handles HTTP routing, middleware, and responses.
    //   express@5 (latest) uses async/await error handling natively.
    "express": "^5.2.1",

    // jsonwebtoken — creates and verifies JSON Web Tokens for authentication.
    //   After login, the server signs a JWT with the user's ID.
    //   The frontend sends this token with every request to prove identity.
    "jsonwebtoken": "^9.0.3",

    // mongoose — ODM (Object Data Modeling) library for MongoDB.
    //   Provides schemas, validation, and query helpers so we work with
    //   JavaScript objects instead of raw MongoDB documents.
    "mongoose": "^9.7.1"
  }
  // NOTE: 'dotenv' and 'nodemon' are intentionally NOT listed here.
  // dotenv is replaced by Node's built-in --env-file flag (Node 20.6+).
  // nodemon is replaced by Node's built-in --watch flag (Node 18+).
}
🔍 Why "type": "module" matters

Without "type": "module", if you write import express from 'express' in a .js file, Node.js will throw: SyntaxError: Cannot use import statement in a module.

With it set, Node treats every .js file in this folder as an ES Module. This means you must also use import for all built-in Node modules: import { createRequire } from 'module', etc.

Frontend — tailwind.config.js

Tailwind CSS scans your source files to find which CSS utility classes you actually use, then generates a minimal CSS bundle containing only those classes. The content array tells Tailwind where to look.

  frontend/tailwind.config.js JavaScript 
// JSDoc type hint — gives your editor IntelliSense for the config object
/** @type {import('tailwindcss').Config} */

// ES Module export (works because frontend uses Vite which supports ESM)
export default {

  // "content" — array of glob patterns pointing to files that USE Tailwind classes.
  // Tailwind reads these files at build time and tree-shakes unused styles.
  // IMPORTANT: If a file is not in this list, its classes won't be in the final CSS.
  content: [
    "./index.html",          // ← root HTML file (Vite's entry point)
    "./src/**/*.{js,ts,jsx,tsx}"  // ← ALL files inside src/ with these extensions
                                //   ** means any subfolder depth
                                //   {js,ts,jsx,tsx} is a brace expansion matching any of those extensions
  ],

  theme: {
    // "extend" — add custom values WITHOUT removing Tailwind's defaults.
    // For example: extend.colors lets you add brand colors while keeping all default colors.
    // In Phase 1 this is empty — we'll add custom tokens in later phases if needed.
    extend: {},
  },

  // "plugins" — official or community Tailwind plugins (e.g., @tailwindcss/forms).
  // Empty for now — no extra plugins needed in this project.
  plugins: [],
}

🔍 How the CSS Pipeline Works

1. Vite starts the dev server and processes main.jsx → imports index.css

2. PostCSS intercepts the CSS and runs the Tailwind plugin

3. Tailwind scans all files matching content globs to find class names

4. Only the found classes are compiled into actual CSS and injected into the browser

5. Result: tiny CSS file with zero unused styles

npm Commands Executed in Phase 1

Every command run during Phase 1, with an explanation of what it does:

  Terminal (PowerShell) Shell 
# ─── BACKEND SETUP ───────────────────────────────────────────────────────

# Creates backend/, config/, middleware/, models/, controllers/, routes/ folders
# -Force: doesn't error if folder already exists
New-Item -ItemType Directory -Force -Path "backend", "backend\config", ...

# Generates backend/package.json with default values (-y skips all questions)
cd backend && npm init -y

# Installs 5 production packages into backend/node_modules/
# Also creates package-lock.json (locks exact dependency versions for reproducibility)
npm install express mongoose jsonwebtoken bcryptjs cors

# Removes nodemon and dotenv that were previously installed
# We replaced them with Node.js built-in --watch and --env-file flags
npm uninstall nodemon dotenv


# ─── FRONTEND SETUP ──────────────────────────────────────────────────────

# Scaffolds a complete React + Vite project in the frontend/ directory
# --template react: uses JavaScript (not TypeScript) React template
# This creates: src/, public/, index.html, vite.config.js, package.json, etc.
npm create vite@latest frontend -- --template react

# Installs all packages listed in frontend/package.json (react, react-dom, vite)
cd frontend && npm install

# Installs Axios (HTTP client) and React Router DOM (client-side routing)
npm install axios react-router-dom

# Installs Tailwind CSS v3, PostCSS, and Autoprefixer as dev dependencies
# (-D is shorthand for --save-dev: these are build tools, not runtime dependencies)
npm install -D tailwindcss@3 postcss autoprefixer

# Initialises Tailwind by generating two config files:
#   tailwind.config.js — where you configure content paths, theme, plugins
#   postcss.config.js  — tells PostCSS to use tailwindcss and autoprefixer plugins
# -p flag: generate postcss.config.js at the same time
npx tailwindcss init -p

Frontend — postcss.config.js

Auto-generated by npx tailwindcss init -p. PostCSS is a CSS processor that runs plugins. This config tells PostCSS which plugins to use when processing CSS files.

  frontend/postcss.config.js JavaScript 
// PostCSS configuration — defines which CSS processing plugins to use
export default {
  plugins: {

    // tailwindcss — the main Tailwind plugin.
    // PostCSS runs this first: it reads tailwind.config.js, scans content files,
    // and replaces @tailwind directives with actual generated CSS classes.
    tailwindcss: {},

    // autoprefixer — runs AFTER tailwindcss.
    // Reads generated CSS and adds vendor-prefixed versions of properties
    // that some browsers still need, e.g.:
    //   display: flex  →  also adds  display: -webkit-flex
    //   transform: ...  →  also adds  -webkit-transform: ...
    // This ensures compatibility with older Safari / iOS browsers.
    autoprefixer: {},
  },
}
